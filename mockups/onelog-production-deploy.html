🔄 4 DATA FLOW CHÍNH — CẦN HIỂU TRƯỚC KHI DEPLOY
1 Log ingest
Client syslog TCP 6514 → Vector (redact PII) → VictoriaLogs. WARN+ tap song song → NATS → Indexer (Drain3) → Qdrant + VL. Retention 7-30d.
2A Chat via WebUI
Browser → Caddy → OpenWebUI → LiteLLM → provider. Model gọi tool → mcpo bridge (MCP→OpenAPI) → mcp-vl/semantic → VL/Qdrant. UI dùng cho team ops không cần Claude Desktop.
2B Chat via Claude Desktop
Claude Desktop → mcp-remote proxy → Caddy /mcp/vl/* + /mcp/semantic/* (Bearer token) → mcp-vl/semantic SSE → VL/Qdrant. KHÔNG đi qua OpenWebUI hay LiteLLM. Model provider là Claude native.
3 Alert + monitor
vmalert eval LogsQL rule mỗi 30s-5m → nếu match → Alertmanager → Telegram. Bao gồm alert DB size (probe Vector 5m/lần).
🔑 SECURITY BOUNDARY
  • • Caddy = single ingress, tất cả internal port bind 127.0.0.1 only
  • • Provider keys tách .env.llm chmod 0400 root — chỉ litellm-proxy đọc
  • • MCP 2-tier token: MCPO_API_KEY (OpenWebUI↔mcpo) + MCP_TOKEN_OPENWEBUI (mcpo↔mcp-*)
  • • PII redact tại Vector trước khi ghi VL (email, IP private, JWT, LLM API key, PEM key)
  • • Backup age-encrypted (public key trên logserver, private key ở ops vault offsite)
📦 PORT EXPOSURE MATRIX
  • • Public 0.0.0.0: 80, 443 (Caddy) · 514/udp, 6514/tcp, 6515/tcp (Vector syslog, limit theo LAN_CIDR)
  • • Loopback 127.0.0.1: 4000 (LiteLLM) · 8080 (agent) · 8090 (OpenWebUI) · 8091 (mcpo) · 9428 (VL) · 8880 (vmalert) · 9093 (Alertmanager)
  • • SSH tunnel only: 8085 (sqlite-web) — KHÔNG expose Internet
  • • Internal Docker network: mcp-vl, mcp-semantic, redis, nats, postgres, qdrant, indexer

Client deploy · 8 bước cho mỗi server nguồn log
chạy trên CLIENT (srv-01, srv-02, ...) ~5 phút / client
Sau khi logserver PASS 15 bước ở trên, wire từng server nguồn log vào. Lab lặp cho <SRV_01_IP> và <SRV_02_IP>. Prod scale ra N server. Script setup-rsyslog-client.sh config rsyslog forward TCP 6514 RFC5424 + queue on-disk 500MB (log không mất khi logserver reboot).
1
SSH vào client + set timezone + verify reachability
MUST timezone khớp logserver (Asia/Saigon) để log timestamp không lệch. Verify TCP 6514 tới logserver mở trước.
# Lab: ssh vietnt@192.168.122.52 · logserver = 192.168.122.53 $ ssh <SSH_USER>@<SRV_01_IP> && sudo timedatectl set-timezone Asia/Saigon && timeout 3 bash -c 'cat </dev/tcp/<LOGSERVER_IP>/6514' && echo " REACHABLE"
Nếu REACHABLE không print → firewall giữa client-logserver chưa mở 6514/tcp. Sửa trước bước tiếp.

6
Fire test log + verify từ logserver
Gửi 1 log tag riêng, chờ 3s, query VL từ logserver. Nếu count = 1 → pipeline OK end-to-end.
# Trên client — fire log $ logger -t onboard-check "verify $(hostname) $(date -Is)"
# Trên logserver — verify (SSH lại logserver hoặc mở terminal khác) $ sleep 3 && curl -sG http://127.0.0.1:9428/select/logsql/query --data-urlencode 'query=service:onboard-check | limit 3' | jq
7
Firewall client-side · outbound 6514/tcp
Nếu client có UFW/iptables outbound rule strict, cần allow explicit. Đa số distro default allow all outbound — bước này chỉ áp dụng khi có policy đặc biệt.
$ sudo ufw status | grep -E 'Status|6514' && sudo ufw allow out to <LOGSERVER_IP> port 6514 proto tcp comment "onelog syslog forward"
8
Lặp cho các client còn lại (loop 1 command)
Sau khi 1 client PASS, deploy hàng loạt bằng loop. SSH key phải configured trước cho tất cả IP (workflow: ssh-copy-id cho mỗi client).
# Chạy trên WORKSTATION — auto SSH + copy + run cho mỗi client $ for IP in <SRV_01_IP> <SRV_02_IP> <SRV_03_IP>; do echo "=== $IP ===" scp ~/onelog/infra/scripts/setup-rsyslog-client.sh <SSH_USER>@$IP:/tmp/ && ssh <SSH_USER>@$IP "sudo LOG_SERVER_IP=<LOGSERVER_IP> bash /tmp/setup-rsyslog-client.sh"; done
✅ VERIFY toàn bộ · TỪ LOGSERVER
# Đếm log mỗi host trong 5 phút qua — mỗi client PHẢI có count > 0 $ curl -sG http://127.0.0.1:9428/select/logsql/query \ --data-urlencode 'query=_time:5m | stats by (host) count() as n | sort by (n desc)' | jq
