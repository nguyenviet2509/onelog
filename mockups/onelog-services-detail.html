Tác dụng chi tiết

  • Giải quyết: 5 ops trước đây phải SSH từng VM tail -f /var/log/* hoặc journalctl → mất thời gian, không correlate được giữa srv-01 và srv-02. VictoriaLogs là "hộp đen trung tâm" gom toàn bộ log JSON-structured về 1 chỗ, query bằng LogsQL.
  • Vị trí trong flow: sink cuối của vector (ghi raw 100% log) · datasource của mcp-vl (Claude query) · datasource của vmalert (rule fire) · backend của VMUI (ops click xem raw).
  • Nếu thiếu: mất khả năng query lịch sử + alerting tức thì. Semantic search vẫn còn (Qdrant template) nhưng không drill xuống được raw line cụ thể → citation rỗng, mất giá trị forensic.
Storage model
Time-partitioned, columnar block. JSON field tự index theo tên. 1 line ~30B sau nén — 5 VM × 24h ≈ vài trăm MB.
Query language
LogsQL: service:nginx _time:1h status:5*. Pipe: | stats by(host) count(). Mạnh hơn grep, dễ hơn ES DSL.
Tiết kiệm
~10% RAM của ELK cùng volume. 512MB cap đủ cho lab 5 VM. Không cần JVM tuning.
Retention
7 ngày default → đổi VL_RETENTION=30d. Compactor tự xoá segment cũ.

  host       : str

Ai write / read
  • • write: indexer upsert sau khi Drain3 + embed
  • • read: mcp-semantic vector search top-k

Tác dụng chi tiết

  • Giải quyết: LogsQL match theo chữ. Khi ops hỏi "có gì bất thường về database không?" mà log thực tế viết InnoDB: Page checksum mismatch → LogsQL bó tay (không có chữ "database"). Qdrant cho phép match theo nghĩa qua vector embedding 1536d.
  • Vị trí trong flow: kho vector duy nhất · indexer upsert (write) · mcp-semantic tìm top-k (read). Hoàn toàn local — không ai khác chạm.
  • Nếu thiếu: mất fuzzy intent path. Ops vẫn dùng được mcp-vl với LogsQL chính xác, nhưng phải biết chính xác keyword → quay lại trải nghiệm "grep" cũ.
Drain3 template
Gom 10⁶ log line / ngày về ~10² template kiểu "User <ID> failed login from <IP>". Embed template, không embed line → giảm 10⁴ cost.
Payload filter
Filter service=mysql AND severity=ERROR chạy trong vector search → vừa semantic vừa precise. Sức mạnh chính của Qdrant.
Cosine distance
Phù hợp embedding OpenAI normalized. Top-k = 5 đủ recall mà vẫn ngắn cho Claude context.
RAM cost
~150MB cho 10⁵ template (1536d float32). Rust no-GC → predictable latency p99 <20ms.

Tác dụng chi tiết

  • Giải quyết: log từ srv-01/02 đi ra dưới nhiều format (syslog RFC3164/5424, journald, app stdout). Vector chuẩn hoá + fan-out: 1 line → song song vào VL (lưu raw) và NATS (filter WARN+ cho semantic). Thay thế cho fluentd/logstash với pipeline gọn 1 file YAML.
  • Vị trí: entry point duy nhất nhận traffic ngoài (514/6514). Đứng trước tất cả storage.
  • Nếu thiếu: không có log nào vào hệ thống. Toàn bộ OneLog tê liệt. Vector là single point of failure → cần HA ở phase 04 (xem docs/ha-roadmap.md).
Fan-out
Sink VL = raw 100%. Sink NATS = filter severity in [WARN,ERROR,CRIT] để tiết kiệm embedding cost.
Disk buffer
/var/lib/vector buffer 256MB. VL/NATS down vẫn nhận syslog → replay khi up. Không mất line.
Enrich
VRL transform thêm field service, host, severity từ regex / hostname → schema VictoriaLogs đồng nhất.
Performance
Rust async. Throughput ~50k lines/s trên 1 vCPU. RAM <100MB ở lab.
