Tổng quan stack

Toàn bộ OneLog là 16 containers trên 1 docker-compose (network ragstack) + 1 client-side drop-in (rsyslog forwarders trên mỗi VM). Chia 6 nhóm: ingestion nhận log, storage lưu trữ, query / mcp phục vụ chat clients, alerting opt-in, llm/chat multi-provider LLM (mới 2026-07), dashboard LLM cost UI (opt-in). Hầu hết bind 127.0.0.1 — chỉ vector :514/6514/6515 và caddy :80/:443 mở ra LAN.

Tác dụng chi tiết

Tiết kiệm

Retention

Ai ghi / đọc

Tác dụng chi tiết

  • Giải quyết: LogsQL match theo chữ. Khi ops hỏi "có gì bất thường về database không?" mà log thực tế viết InnoDB: Page checksum mismatch → LogsQL bó tay (không có chữ "database"). Qdrant cho phép match theo nghĩa qua vector embedding 1536d.
  • Vị trí trong flow: kho vector duy nhất · indexer upsert (write) · mcp-semantic tìm top-k (read). Hoàn toàn local — không ai khác chạm.
  • Nếu thiếu: mất fuzzy intent path. Ops vẫn dùng được mcp-vl với LogsQL chính xác, nhưng phải biết chính xác keyword → quay lại trải nghiệm "grep" cũ.
Template Drain3
Gom 10⁶ log line / ngày về ~10² template kiểu "User <ID> failed login from <IP>". Embed template, không embed line → giảm 10⁴ chi phí.
Payload filter
Filter service=mysql AND severity=ERROR chạy trong vector search → vừa semantic vừa precise. Sức mạnh chính của Qdrant.
Cosine distance
Phù hợp embedding OpenAI normalized. Top-k = 5 đủ recall mà vẫn ngắn cho Claude context.
Chi phí RAM
~150MB cho 10⁵ template (1536d float32). Rust no-GC → latency ổn định p99 <20ms.

ECS-lite schema (Option B — required)
{
  "@timestamp": "2026-06-25T09:06:55+07:00",   # RFC3339
  "host":    { "name": "srv-02" },
  "log":     { "level": "error" },             # info|warn|error|...
  "service": { "name": "mysql" },
  "message": "Too many connections (max=151)",
  "labels":  { "env": "prod", "az": "lab" },   # optional pass-through
  "trace":   { "id": "..." }                   # optional
}
Unknown top-level fields bị drop ở Vector rsyslog_json_normalize — anti schema-drift.
rsyslog disk-assisted queue (cả 2 option)
queue.type="LinkedList"
queue.size="10000"
queue.filename="onelog_json_fwd"
queue.saveOnShutdown="on"
queue.maxDiskSpace="500m"
action.resumeRetryCount="-1"   # retry forever
Logserver hoặc network down → rsyslog buffer ra disk (max 500MB/client), replay khi server up. Tương đương disk buffer của Vector phía server — defence in depth.

Tác dụng chi tiết

  • Giải quyết: Vector listen :6514/:6515 nhưng không tự đến lấy log ở client. Phải có agent chạy trên từng client VM push log sang. rsyslog là agent có sẵn trên mọi Linux distro → KISS, không cài thêm gì. 2 option vì 2 use-case: greenfield (text) hoặc đã có JSON pipeline (structured).
  • Vị trí: trên srv-01 / srv-02 / mọi client tương lai. Đẩy TCP → vector trên logserver. Là biên ngoài cùng của OneLog data plane.
  • Nếu thiếu: Vector chỉ có log từ mock-logs.py (UDP 514 local). Không có log thật từ client → toàn bộ stack chạy không tải.
Hợp đồng schema
Option B bắt buộc 5 field @timestamp, host.name, log.level, service.name, message. Thiếu = Vector drop / VL field rỗng.
Nested qua constant()
rsyslog template không hỗ trợ nested JSON natively → ghép bằng constant("\"host\":{\"name\":\"") + property(format="json").
Không TLS · không auth (PoC)
Plain TCP. Bắt buộc firewall whitelist client IP trên logserver. TLS + token = plan riêng (roadmap).
Xử lý sự cố
ss không ESTAB → firewall / IP sai. rsyslogd -N1 báo jsonf → rsyslog <8.x, dùng Option A. _msg rỗng ở VL → escape quote trong template.
Dev / smoke test dùng mock-logs.py + mock-logs.service (UDP 514 local, không cần client thực). Production: chọn 1 option Đối với mỗi client — đừng bật cả 2 cùng lúc (duplicate ingestion).

Volumes

Pipeline (vector.yaml)
sources:
  syslog_udp:        {type: syslog, mode: udp, :514}
  syslog_tcp:        {type: syslog, mode: tcp, :6514}
  rsyslog_json_tcp:  {type: socket, mode: tcp, :6515,
                      decoding: json,
                      framing: newline_delimited}

transforms:
  enrich:                  # syslog → flat schema
    inputs: [syslog_udp, syslog_tcp]
    .service/.host/.severity/._msg/._time
  rsyslog_json_normalize:  # ECS-lite → flat schema (NEW)
    inputs: [rsyslog_json_tcp]
    whitelist @timestamp,host.name,log.level,
              service.name,message,labels.*,trace.*
  to_warn: filter severity in [WARN,ERROR,CRIT]

sinks:
  victoria_logs: → http://vl:9428  (raw 100%)
  nats_warn:     → subject logs.warn  (filter)

Tác dụng chi tiết

  • Giải quyết: log từ srv-01/02 đi ra dưới nhiều format (syslog RFC3164/5424, journald, app stdout). Vector chuẩn hoá + fan-out: 1 line → song song vào VL (lưu raw) và NATS (filter WARN+ cho semantic). Thay thế cho fluentd/logstash với pipeline gọn 1 file YAML.
  • Vị trí: entry point duy nhất nhận traffic ngoài (514/6514). Đứng trước tất cả storage.
  • Nếu thiếu: không có log nào vào hệ thống. Toàn bộ OneLog tê liệt. Vector là single point of failure → cần HA ở phase 04 (xem docs/ha-roadmap.md).

:514/udp mock-logs · :6514/tcp rsyslog text (Opt A) · :6515/tcp rsyslog JSON ECS-lite (Opt B).
Fan-out
Sink VL = raw 100%. Sink NATS = filter severity in [WARN,ERROR,CRIT] để tiết kiệm chi phí embedding.
